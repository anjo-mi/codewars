Mixins</a></h2><h3>7 kyu</h3><h1 id="learning-typescript-mixins">Learning TypeScript. Mixins</h1><h2 id="overview">Overview</h2><p>Along with traditional object-oriented hierarchies, another popular way of building up classes from reusable components is to build them by combining simpler partial classes. You may be familiar with the idea of mixins or traits for languages like Scala, and the pattern has also reached some popularity in the JavaScript community.</p><h2 id="introduction-to-mixins">Introduction to mixins</h2><p>Generally speaking a mixin class is a class that implements a distinct aspect of functionality. Other classes can then include the mixin and access its methods and properties. That way, mixins provide a form of code reuse that is based on composing behavior.</p><p>Now let’s see how you can model mixins in TypeScript. First of all we’ll define two classes <code>Disposable</code> and <code>Activatable</code> that will act as our mixins. You can see each one is focused on a particular activity or capability. We’ll later mix these together to form a new class from both capabilities.</p><pre><code class="language-typescript"><span class="cm-comment">/** Disposable Mixin. */</span><span class="cm-keyword">class</span> <span class="cm-def">Disposable</span> {    <span class="cm-property">isDisposed</span>: <span class="cm-type">boolean</span>;    <span class="cm-property">dispose</span>() {        <span class="cm-keyword">this</span>.<span class="cm-property">isDisposed</span> <span class="cm-operator">=</span> <span class="cm-atom">true</span>;    }}<span class="cm-comment">/** Activatable Mixin. */</span><span class="cm-keyword">class</span> <span class="cm-def">Activatable</span> {    <span class="cm-property">isActive</span>: <span class="cm-type">boolean</span>;    <span class="cm-property">activate</span>() {        <span class="cm-keyword">this</span>.<span class="cm-property">isActive</span> <span class="cm-operator">=</span> <span class="cm-atom">true</span>;    }    <span class="cm-property">deactivate</span>() {        <span class="cm-keyword">this</span>.<span class="cm-property">isActive</span> <span class="cm-operator">=</span> <span class="cm-atom">false</span>;    }}</code></pre><p>Next, we’ll create the class that will handle the combination of the two mixins. Let’s look at this in more detail to see how it does this:</p><pre><code class="language-typescript"><span class="cm-keyword">class</span> <span class="cm-def">SmartObject</span> <span class="cm-keyword">implements</span> <span class="cm-type">Disposable</span>, <span class="cm-type">Activatable</span> {</code></pre><p>The first thing you may notice in the above is that instead of using extends, we use implements. This treats the classes as interfaces, and only uses the types behind Disposable and Activatable rather than the implementation. This means that we’ll have to provide the implementation in class. Except, that’s exactly what we want to avoid by using mixins.</p><p>To satisfy this requirement, we create stand-in properties and their types for the members that will come from our mixins. This satisfies the compiler that these members will be available at runtime. This lets us still get the benefit of the mixins, albeit with some bookkeeping overhead.</p><pre><code class="language-typescript"><span class="cm-comment">// Disposable</span><span class="cm-variable">isDisposed</span>: <span class="cm-variable">boolean</span> <span class="cm-operator">=</span> <span class="cm-atom">false</span>;<span class="cm-variable">dispose</span>: () <span class="cm-operator">=&gt;</span> <span class="cm-keyword">void</span>;<span class="cm-comment">// Activatable</span><span class="cm-variable">isActive</span>: <span class="cm-variable">boolean</span> <span class="cm-operator">=</span> <span class="cm-atom">false</span>;<span class="cm-variable">activate</span>: () <span class="cm-operator">=&gt;</span> <span class="cm-keyword">void</span>;<span class="cm-variable">deactivate</span>: () <span class="cm-operator">=&gt;</span> <span class="cm-keyword">void</span>;</code></pre><p>Finally, we mix our mixins into the class, creating the full implementation.</p><pre><code class="language-typescript"><span class="cm-variable">applyMixins</span>(<span class="cm-variable">SmartObject</span>, [<span class="cm-variable">Disposable</span>, <span class="cm-variable">Activatable</span>]);</code></pre><p>Lastly, we create a helper function that will do the mixing for us. This will run through the properties of each of the mixins and copy them over to the target of the mixins, filling out the stand-in properties with their implementations.</p><pre><code class="language-typescript"><span class="cm-keyword">function</span> <span class="cm-def">applyMixins</span>(<span class="cm-def">derivedCtor</span>: <span class="cm-type">any</span>, <span class="cm-def">baseCtors</span>: <span class="cm-type">any</span>[]) {    <span class="cm-variable-2">baseCtors</span>.<span class="cm-property">forEach</span>(<span class="cm-def">baseCtor</span> <span class="cm-operator">=&gt;</span> {        <span class="cm-variable">Object</span>.<span class="cm-property">getOwnPropertyNames</span>(<span class="cm-variable-2">baseCtor</span>.<span class="cm-property">prototype</span>).<span class="cm-property">forEach</span>(<span class="cm-def">name</span> <span class="cm-operator">=&gt;</span> {            <span class="cm-variable-2">derivedCtor</span>.<span class="cm-property">prototype</span>[<span class="cm-variable-2">name</span>] <span class="cm-operator">=</span> <span class="cm-variable-2">baseCtor</span>.<span class="cm-property">prototype</span>[<span class="cm-variable-2">name</span>];        });    });}</code></pre><p><b>Hint:</b> You can find the complete source code for this sample here: <a href="https://www.typescriptlang.org/docs/handbook/mixins.html" data-turbolinks="false" target="_blank">https://www.typescriptlang.org/docs/handbook/mixins.html</a></p><h2 id="task">Task</h2><p>You're required to implement mixin <code>Serializable</code> which should contain:</p><ul><li>method <code>public serialize(): string</code> - this method must serialize all properties of the object (i.e. return object converted to string).</li><li>method <code>public deserialize(source: string): void</code> - this is the opposite method, that converts string to object properties and assigns them to current object.</li></ul><p>For simplicity's sake let's consider that objects won't contain any circular references.The function <code>applyMixins</code> is defined in tests and you don't need to change it.</p><p><strong>P.S.</strong> Solved this kata?
